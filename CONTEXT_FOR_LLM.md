# Context for LLM — Fonction Labs BP Pipeline v2

## Purpose

This repository contains a reproducible financial model pipeline for Fonction Labs' seed fundraising materials.

The goal is to maintain a single source of truth for:

- the full BP workbook;
- a simplified VC-facing workbook;
- a static HTML dashboard;
- validation checks.

The pipeline should avoid any manual divergence between Excel, dashboard and narrative outputs.

## Company context

Fonction Labs is positioning itself as the **Agent OS for Enterprise Operations**.

The core thesis:

- enterprise work is shifting from human-operated workflows to agentic operations;
- AI agents can execute work, but companies need the operating layer to deploy, supervise, audit and improve them;
- Fonction starts with enterprise deployments and then compounds these patterns into recurring platform revenue.

The model is forward-deployed, inspired by Palantir / Sierra, but focused on backend / enterprise operations rather than only customer service.

## Key strategic constraints from the discussion

1. The BP must show that Fonction already generates revenue.
2. The model must show the transition from services / deployment revenue to recurring platform revenue.
3. Services are not hidden; they are an entry step, but the long-term value comes from subscriptions per live use case.
4. BPCE / In Curve / Wassym should remain in the model detail but should not appear as a named line in VC-facing dashboard outputs.
5. The dashboard must be VC-facing: no Qonto / BPCE / Wassym / internal accounting names.
6. The full BP is a model output generated from raw data + assumptions.
7. The simplified BP and dashboard must be views derived from the same model tables.
8. All figures must be assumptions or calculated outputs, not arbitrary manual numbers.
9. Use cases and clients must be displayed as integers in VC-facing outputs.
10. Ending ARR means annualized recurring revenue run-rate at period end: `live use cases × subscription MRR × 12`.

## Core model assumptions

Defined in `data/assumptions/vc_case.yaml`.

Key pricing assumptions:

- Discover / workshop: **10k€** per new enterprise client.
- Deploy / 0→1 deployment: **40k€** per use case, recognized over 3 months.
- Run & Scale / subscription: **10k€ MRR** per live use case.
- FDE support: **1,150€/day** when explicitly modeled.
- Usage / success-based pricing is deliberately set to zero for now; it may become a later upside model.

FDE capacity assumptions:

- 2026: **2 active deployment use cases per FDE**.
- 2027: **3 active deployment use cases per FDE**.
- 2028: **4 active deployment use cases per FDE**.

Improvement is explained by reusable playbooks, internal tooling, connectors, evaluation flows and progressive self-serve deployment capabilities.

## Pipeline architecture

```text
raw data + assumptions
→ normalized tables
→ DuckDB model tables
→ full BP Excel
→ simplified BP Excel
→ dashboard_data.js / dashboard HTML
→ validation
```

## Source of truth

`data/processed/model.duckdb` is the model database generated by the pipeline. Excel and HTML are outputs.

Do not treat manually edited workbooks as source of truth.

## Validation checks

`validate.py` checks:

- Jan-Jun 2026 commercial revenue matches the target actuals.
- Dec-2027 ending ARR matches the target check.
- Dec-2028 ending ARR matches the target check.
- Ending ARR equals `live_use_cases × subscription_mrr × 12`.
- Annual revenue equals the monthly sum.
- Dashboard KPIs align with model tables.
- Full workbook has expected sheets.

## VC-facing wording constraints

Avoid:

- Qonto;
- BPCE;
- Wassym;
- In Curve;
- legacy services;
- real receipts;
- defensive phrasing like “not services versus software”.

Use:

- Actual commercial revenue;
- Services & Deployment Revenue;
- Platform Subscription Revenue;
- Enterprise accounts;
- Live use cases;
- Ending ARR;
- FDE leverage;
- Use of funds;
- Runway.

## Known limitations / next steps

- Attio data is currently represented by a funnel extract, not a complete CRM export.
- The Qonto parser uses classification rules defined in assumptions; these should be reviewed before investor distribution.
- Costs for cold callers, sales full-time, product lead and FDE profiles include placeholders that should be confirmed.
- Additional scenarios (`base_case.yaml`, `upside_case.yaml`) can be added using the same pipeline.

## Dashboard regression fix

A later pipeline v2 dashboard was too short and visually regressed versus the earlier hybrid dashboard. The corrected package keeps the pipeline architecture but restores a fuller dashboard-only view with stronger visual hierarchy, more complete sections, larger charts, year-by-year trajectory cards, and explicit download links. The dashboard remains a derived view from `dashboard_data.js`; the model source of truth remains `model.duckdb` and `model_outputs.json`.
